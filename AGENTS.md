# AGENTS.md — спека для ИИ-агента

Это инструкция для тебя, ИИ-кодинг-агента (Claude Code, Cursor и т.д.),
работающего в этом репозитории на стороне сервера. Прочитай весь файл
до того, как начнёшь править код.

## 1. Кто пользователь

Пользователь — **участник вайбкодинг-мероприятия**. У него:

- нет доступа к терминалу;
- нет git, docker, kubectl, IDE;
- единственный канал общения с тобой — этот чат;
- он формулирует, что ему нужно, своими словами на русском
  («сделай экран с расписанием», «добавь кнопку поделиться»);
- он не знает технических подробностей и **не должен их знать**.

Поэтому:

- **Не задавай технических вопросов** («какой PostgreSQL поставить?»,
  «React или Svelte?», «оставить TypeScript строгим?»). Решай сам,
  выбирай разумные дефолты, пиши участнику только результат.
- **Если запрос неоднозначный**, переспроси один раз
  и только в категориях «функция / визуал / содержание». Пример:
  «Сделать кнопкой синего цвета или зелёного?» — ОК,
  «Использовать Zustand или Redux?» — НЕ ОК.
- **Отвечай на русском**, кратко, без жаргона.

## 2. Цикл работы

Каждый запрос участника обрабатывается так:

1. Понять, что нужно поменять.
2. Внести правку в `app/src/`.
3. Запустить `npm run typecheck` в `app/` — должен проходить.
4. Опубликовать через
   ```
   APP_SLUG=$APP_SLUG ./scripts/deploy.sh
   ```
   (переменная `APP_SLUG` уже выставлена в окружении дев-контейнера —
   она же в имени namespace.)
5. Дождаться, пока `kubectl rollout status` отдаст success.
6. Ответить участнику ссылкой
   `https://ranepa.gigaparsers.ru/$APP_SLUG/` и одним-двумя
   предложениями про то, что изменилось.

Если шаг 3 или 5 упал — **не отвечай участнику «готово»**. Сначала
почини. Логи: `kubectl -n ranepa-$APP_SLUG logs deploy/ranepa-app`,
`kubectl -n ranepa-$APP_SLUG describe deploy/ranepa-app`.

## 3. Архитектура репозитория

```

│   ├── App.tsx           # корневой компонент, switch по route
│   ├── main.tsx          # bootstrap React'а
│   ├── routes/           # экраны: Home / List / Detail (примеры)
│   ├── components/       # переиспользуемые: Screen, Button, Card
│   ├── lib/              # router, storage (LocalStorage)
│   └── styles/
│       ├── main.css      # Tailwind + базовые стили
│       └── brand.css     # ЦВЕТА БРЕНДА — менять только эти переменные
├── public/
│   ├── manifest.webmanifest    # имя/иконки PWA
│   └── brand/                  # логотип, favicon, apple-touch-icon
├── index.html
├── vite.config.ts        # base = process.env.APP_BASE — НЕ ТРОГАТЬ
└── package.json

infra/                    # инфра, НЕ менять без явной просьбы
├── Dockerfile
├── nginx.conf
├── edge-nginx/

scripts/
```

## 4. Что МОЖНО менять

Свободно меняй:

- **`app/src/routes/*`** — экраны. Создавай новые файлы здесь.
- **`app/src/components/*`** — переиспользуемые UI-блоки.
- **`app/src/lib/*`** — утилиты (хранение, роутер, форматтеры).
- **`app/src/App.tsx`** — добавляй ветки в switch для новых экранов
  (и id в `Route` в `app/src/lib/router.ts`).
- **`app/src/styles/brand.css`** — меняй цвета бренда, если участник
  попросил «сделай зелёным».
- **`app/public/brand/*`** — заменяй логотип/иконки, если есть новый.
- **`app/index.html`** — `<title>`, `<meta name="apple-mobile-web-app-title">`.
- **`app/public/manifest.webmanifest`** — `name`, `short_name`, `theme_color`.

## 5. Что НЕ ТРОГАТЬ

Если нет явной просьбы — не редактируй:

- **`infra/*`** — Dockerfile, nginx.conf, k8s манифесты. Они работают.
- **`scripts/*`** — кроме случая, когда участник просит «сделай так,
  чтобы билд был быстрее» (тогда сначала спроси меня).
- **`vite.config.ts` строка `base = process.env.APP_BASE`** — если
  её сломать, приложение не откроется по подпути `/app1/`.
- **Корневой `package.json`** (на уровне репы) — все JS-зависимости
  живут только в `app/package.json`.

Не делай:

- Не добавляй state-менеджеры (Redux/Zustand/Jotai) пока приложение
  не разрослось до >5 экранов с разделяемым состоянием.
- Не подключай UI-библиотеки (MUI/AntD/Chakra) — у нас Tailwind
  и собственные `Button`/`Card`/`Screen`. Если нужен компонент,
  которого нет — напиши его в `components/`, не тащи библиотеку.
- Не подключай маршрутизаторы (react-router) — `useRouter` в `lib/`
  справляется. Если экранов станет >7 — тогда подумаем.
- Не добавляй CSS-фреймворки помимо Tailwind.

## 6. Стиль кода

- **TypeScript строгий**, без `any`. Если тип неизвестен — `unknown`
  с разбором.
- **Компоненты ≤ 200 строк**. Если разрастается — разбивай.
- **Tailwind**, без inline-`style={{...}}` (исключение — динамические
  значения, которые невозможно выразить классами).
- **Имена**: компоненты PascalCase, файлы PascalCase.tsx, утилиты
  camelCase.ts.
- **Без комментариев типа `// добавил кнопку`** — git diff и так
  показывает. Комментарий нужен только если поведение неочевидно.
- **Без библиотек дат/чисел**, пока штатных API не хватает.
- **Без Telegram WebApp / Telegram-логики** — это шаблон обычной
  веб-эппы (хоть и с iOS-look). Если участник попросит Telegram-бота,
  это отдельная история, скажи мне.

## 7. iOS-look — обязательные элементы

Каждый новый экран должен:

- использовать компонент `<Screen>` как корень;
- уважать safe-area (классы `pt-safe-top`, `pb-safe-bottom`);
- использовать токены бренда `var(--brand-*)`, не хардкодить цвета;
- иметь тапаемые элементы ≥ 44pt по высоте (`h-11` ≈ 44px);
- работать в light и dark теме (тестируется через
  `prefers-color-scheme` — токены в `brand.css` уже это умеют).

## 8. Как добавить новый экран

Пример: участник просит «добавь экран профиля».

1. `app/src/lib/router.ts` — добавь `"profile"` в `Route`:
   ```ts
   export type Route = "home" | "list" | "detail" | "profile";
   ```
2. `app/src/routes/Profile.tsx`:
   ```tsx
   import { Screen } from "../components/Screen";

   type Props = { onBack: () => void };

   export function Profile({ onBack }: Props) {
     return (
       <Screen title="Профиль" onBack={onBack}>
         <div className="pt-3">…</div>
       </Screen>
     );
   }
   ```
3. `app/src/App.tsx` — добавь ветку:
   ```tsx
   case "profile":
     return <Profile onBack={() => navigate("home")} />;
   ```
4. Откуда туда переходить — добавь кнопку в `Home.tsx` или где
   попросил участник.

## 9. Как добавить ручку API

По умолчанию шаблон **без бэкенда** (статика + LocalStorage).
Если участник попросил «чтобы данные сохранялись на сервере» или
«чтобы был ИИ-чат» — нужен бэкенд. Прежде чем поднимать FastAPI,
спроси меня. Не разводи бэкенд молча.

## 10. Как задеплоить

```sh
APP_SLUG=$APP_SLUG ./scripts/deploy.sh
```

Если упало:

```sh
kubectl -n ranepa-$APP_SLUG describe deploy/ranepa-app
kubectl -n ranepa-$APP_SLUG logs deploy/ranepa-app --tail=200
```

Откатить деплой к предыдущей версии:

```sh
kubectl -n ranepa-$APP_SLUG rollout undo deploy/ranepa-app
```

## 11. Если всё совсем сломалось

`./scripts/reset.sh` — вернёт код к `origin/main` (то, что было в
шаблоне в начале мероприятия). После него всё равно нужен
`./scripts/deploy.sh`, чтобы откатить и опубликованную версию.

## 12. Инструмент: распознавание изображения через VLM

Если участник прислал картинку (скриншот, фото этикетки, схему,
чертёж, документ) или просит сделать функцию, требующую понимания
изображения («сканер штрихкодов», «распознай товар по фото»,
«извлеки данные из чека») — используй `./scripts/vlm.sh`:

```sh
./scripts/vlm.sh <путь-к-файлу> "что нужно извлечь / описать"
```

Примеры:

```sh
./scripts/vlm.sh /tmp/label.jpg "извлеки штрихкод и название товара"
./scripts/vlm.sh /tmp/schedule.png "распиши расписание текстом, день/время/занятие"
./scripts/vlm.sh /tmp/screenshot.png "что тут изображено и какие элементы UI"
```

Скрипт идёт через GigaChat-Vision (модель GigaChat-2-Max). Поддерживает
JPEG/PNG/GIF/WebP. На stdout — текст ответа модели; в случае ошибки —
её тело в stderr.

Когда применять:
- Участник кидает скриншот и просит сделать «такой же экран» —
  vlm.sh даст текстовое описание элементов, на основе которого
  ты сгенерируешь JSX/CSS.
- Участник просит фичу «сканер X» — vlm.sh даст proof-of-concept
  распознавания, чтобы ты понимал что вернёт API на типичных входах.
- Любой запрос «что на картинке» — сразу через vlm.sh, не пытайся
  гадать.

Скрипт работает независимо от провайдера твоего слота (Claude /
DeepSeek / GigaChat) — обращается к выделенному gpt2giga-сервису.
